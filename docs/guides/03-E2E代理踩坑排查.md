## 1. 问题背景

E2E 测试需要 vite `server.proxy` 根据请求头 `X-Backend` 动态转发到不同镜像源（如 `https://hj260602947.top`），与生产 `worker.ts` 行为对齐。

## 2. 踩坑点

### 坑 1：Vite 6 的 `router` 选项不生效

**现象**：按 Vite 文档配置 `router: (req) => u.origin`，请求仍走默认 `target`，连接 `haijiao.com` 超时。

**原因**：Vite 6.4.3 内部使用 `http-proxy-3`（非原版 `http-proxy`），`router` 选项未被正确传递给 http-proxy。Vite 源码中 `proxy.web(req, res, options)` 传入空 `options`，而 `router` 是在 `createProxyServer` 时解析的，不会在每次请求时重新调用。

**结论**：Vite 6 的 `server.proxy.router` 选项**不能**用于动态代理。

---

### 坑 2：`configure` + `proxyReq` 修改 `options.target` 无效

**现象**：在 `configure` 中监听 `proxyReq` 事件修改 `options.target`，请求仍走旧 target。

**原因**：`proxyReq` 事件在 TCP 连接到目标服务器**之后**才触发。此时连接已建立，修改 `options.target` 只影响**下一个**请求，当前请求仍走原 target。

**关键时序**：
```
proxy.web(req, res) 
  → 读取 options.target 建立 TCP 连接 
  → 发送请求到目标 
  → 触发 proxyReq 事件（此时 target 已确定，修改无效）
```

---

### 坑 3：GET 请求携带 `Content-Type: application/json` 导致 500

**现象**：代理正确转发到镜像源，但后端返回 500 Internal Server Error。

**原因**：`request.ts` 的 `getAuthHeaders()` 对所有请求（包括 GET）统一设置了 `Content-Type: application/json`。后端对 GET 请求收到 `Content-Type` 头时返回 500。

**修复**：只在有请求体（POST/PUT）时才设置 `Content-Type`。

```ts
// 修复前
const headers = {
  'Content-Type': 'application/json',  // ❌ GET 请求不应带此头
  'X-Backend': settings.apiBase,
}

// 修复后
const headers: Record<string, string> = {
  'X-Backend': settings.apiBase,
  ...(body ? { 'Content-Type': 'application/json' } : {}),
}
```

---

### 坑 4：`rewrite` 去掉 `/api` 前缀导致后端返回 HTML

**现象**：配置 `rewrite: (p) => p.replace(/^\/api/, '')` 后，请求返回 HTML 页面而非 JSON。

**原因**：`worker.ts` 中会去掉 `/api` 前缀再转发到后端，但**本地代理直接转发到镜像源**，镜像源的 API 路径需要保留 `/api` 前缀。

```
worker.ts 行为：  /api/topic/hot  →  去掉 /api  →  ${backend}/topic/hot  ✅
vite proxy 行为：  /api/topic/hot  →  rewrite 去 /api  →  ${backend}/topic/hot  ❌
```

**关键区别**：Worker 是后端中间层，会重写路径；vite proxy 是透明代理，路径原样转发。

**修复**：删除 `rewrite` 配置，保留 `/api` 前缀。

---

### 坑 5：最终方案 — 自定义中间件替代 `server.proxy`

**原因总结**：Vite 6 的 `server.proxy` 不支持真正的动态代理（`router` 不生效、`configure` 时序问题），无法满足根据 `X-Backend` 头动态转发的需求。

**解决方案**：通过 Vite 插件的 `configureServer` 钩子注入自定义中间件，直接用 `node:https` 模块发请求：

```ts
function dynamicProxyPlugin() {
  return {
    name: 'dynamic-proxy',
    configureServer(server: any) {
      server.middlewares.use((req: any, res: any, next: any) => {
        if (!req.url?.startsWith('/api')) return next()
        const backend = (req.headers['x-backend'] as string) || ''
        if (!backend) return next()
        let target: URL
        try {
          target = new URL(backend)
          if (target.protocol !== 'https:') return next()
        } catch { return next() }

        const proxyReq = https.request({
          hostname: target.hostname,
          port: 443,
          path: req.url,
          method: req.method,
          headers: { ...req.headers, host: target.host },
          rejectAuthorized: false,
        }, (proxyRes) => {
          res.writeHead(proxyRes.statusCode || 502, proxyRes.headers)
          proxyRes.pipe(res)
        })
        proxyReq.on('error', (err) => {
          res.writeHead(502)
          res.end('Proxy Error')
        })
        req.pipe(proxyReq)
      })
    },
  }
}
```

**优点**：
- 完全控制代理行为，每请求独立处理
- 无 Vite 版本兼容性问题
- 仅 `npm run dev` 生效，不影响生产构建

## 5. 经验总结

1. **不要假设 Vite 选项等同于底层库选项**：Vite 对 `http-proxy` 做了封装，部分选项（如 `router`）可能不被正确传递
2. **`proxyReq` 事件不能修改 target**：事件触发时连接已建立，只能影响后续请求
3. **GET 请求不应携带 `Content-Type`**：某些后端服务器会因此返回 4xx/5xx
4. **本地代理与生产代理路径处理不同**：Worker 会重写路径，vite proxy 是透明转发
5. **自定义中间件是最可靠的方案**：当内置代理不满足需求时，直接写中间件比 hack 内置选项更可控

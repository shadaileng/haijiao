## 1. 问题描述

帖子 `/topic/1742505` 的视频在当前应用无法播放，但在原网站（`https://hj2606029d3.top/post/details?pid=1742505`）可以正常播放。

## 2. 根因分析

### 2.1 排除项

| 怀疑点 | 排查结果 |
|:-------|:---------|
| `resolveRealUrl` 条件判断 | 视频为免费帖（`sale: null`），URL 为 `_i.m3u8`（完整视频，非 preview），条件不相关 |
| `sale` 传入不正确 | 视频为免费帖，`sale` 不存在，流程正确 |
| preview URL 格式 | URL 为 `_i.m3u8`，`resolveRealUrl` 是 no-op |
| 附件列表缺少 video | API 正常返回附件 `id: 10620278, category: video` |

### 2.2 真实根因：自定义 AES-128 密钥包装 (Key Wrapping)

原网站使用 WebAssembly 实现的 **AES-128 密钥包装** 作为自定义 DRM 层：

#### 加密链路

```
m3u8 声明 #EXT-X-KEY:METHOD=AES-128,URI="enc_10620278.key"
                  │
enc_10620278.key ─┤ (16 字节，并非真实 AES 密钥，而是被包装密钥)
                  │
                  ├── jquery.wasm ── jquery_key(encrypted_key, 16, wrapping_key, len)
                  │                     │
                  │                     └─> 解密得到真实 AES 密钥
                  │
                  └── {m3u8}.jpg ── atob(base64) ── UTF-8 bytes ── wrapping_key
```

1. **m3u8 URL** `.../1062027852nihTXU_i.m3u8`
2. **密钥文件** `.../enc_10620278.key` — 16 bytes 被包装的密钥，非原始 AES 密钥
3. **包装密钥 URL** `.../1062027852nihTXU_i.jpg` — 替换 m3u8 的 `.m3u8` → `.jpg`，内容为 base64 编码的 MongoDB 连接字符串 `mongodb://admin:S3cUr3_P4ss_2026@6.90.10.19:27017/auth`
4. **WASM 模块** `/js/jquery.wasm` (12825 bytes)，Emscripten 编译，导出 `jquery_key` 函数
5. **解密流程**：`jquery_key(encrypted_key_ptr, 16, wrapping_key_ptr, wrapping_key_len)` — 用包装密钥解密被包装密钥，得到真实 AES-128 密钥，修改内存原地

#### Node.js 破解验证

```javascript
// 1. 获取被包装密钥
encryptedKey = d10f2bdd7661ce5415d6d615c2afd79d (16 bytes)

// 2. 获取包装密钥 (.jpg → atob → UTF-8)
wrappingKey = "mongodb://admin:S3cUr3_P4ss_2026@6.90.10.19:27017/auth" (54 bytes UTF-8)

// 3. WebAssembly jquery_key 解密
jquery_key(encPtr, 16, wrapPtr, 54)
→ decryptedKey = bc6045ba1905ac6e3af9b771afc6b9a7 (16 bytes) ✅

// 4. 用解密后的 AES 密钥解密 TS 片段
const decipher = createDecipheriv('aes-128-cbc', decryptedKey, iv)
const decrypted = decipher.update(encryptedTS) + decipher.final()
→ 首字节 0x47 (MPEG-TS sync) ✅
→ h264 startcode 在 offset 199 ✅
→ ffmpeg 可播放 ✅
```

#### 原网站代码参考

在 `app.455ca12d.js` 中包含三套几乎相同的 HLS 初始化逻辑（对应不同视频播放场景），核心流程：

```javascript
// 1. 获取包装密钥
var wrapKeyUrl = m3u8Url.replace(".m3u8", ".jpg");
fetch(wrapKeyUrl)
  .then(r => r.text())
  .then(text => {
    var decoded = atob(text.trim());
    wrapKey = new Uint8Array(decoded.length);
    // 填充字节
  });

// 2. 加载 WASM
var wasmModule = createModule().then(instance => { wasm = instance; });

// 3. 初始化 HLS
Promise.all([wrapKeyPromise, wasmPromise]).then(() => {
  var hls = new Hls({ autoStartLoad: false, ... });
  hls.on(Hls.Events.KEY_LOADED, (event, data) => {

    wasm.exports.jquery_key(encPtr, 16, wrapPtr, wrapKey.length);

  hls.attachMedia(video);
  hls.startLoad();

```

**WasmKeyLoader**（内联类，`content.ts:311-337`）：

```typescript
class WasmKeyLoader {
  load(context, config, callbacks) {
    if (context.frag?.decryptdata && context.url?.includes('.key')) {
      fetch(context.url)           // 获取被包装密钥
        .then(r => r.arrayBuffer())
        .then(buf => {
          const decrypted = decryptWasmKey(wasm, wrapKey, buf)  // WASM 解密
          callbacks.onSuccess({ url, data: decrypted }, ...)     // 返回真实密钥
        })
    } else {
      this.defaultLoader.load(context, config, callbacks)
    }
  }
}
```

## 4. 浏览器兼容问题（待解决）

### 4.1 当前状态

- ✅ Node.js 端：WASM 解密正确 → TS 片段可播放
- ✅ 现有基础设施已就绪（`getWasmInstance` / `getWrapKey` / `decryptWasmKey`）
- ❌ HLS.js v1.6.16 浏览器端：自定义 key loader 返回解密密钥后，transmuxer 处理失败

### 4.2 尝试过的方案

| 方案 | 结果 |
|:-----|:-----|
| `KEY_LOADED` 事件中修改 `frag.decryptdata.key` | ❌ transmuxer-interface push error |
| 使用原站 hls.js v0.13.2（`hls.legacy.min.js`） | ❌ internalException FATAL |
| 自定义 loader 拦截密钥请求返回解密后的密钥 | ❌ manifest 解析成功但片段仍未播放 |

### 4.3 可能的原因

1. **HLS.js v1.6.16 transmuxer 兼容问题** — 新版 transmuxer 运行在 WebWorker 中，解密/密钥传递链路与旧版不同
2. **TS 片段结构** — 该视频流为纯 h264（无音频），新版 transmuxer 片段校验更严格
3. **loader 回调签名不匹配** — 自定义 loader 的 `onSuccess` 参数与 HLS.js 内部期望的 `LoaderCallbacks` 不完全一致，可能导致密钥未正确注册

## 5. 解决方案

| 方案 | 概述 | 改动量 | 风险 | 备注 |
|:----:|:-----|:------:|:----:|:-----|
| **方案一 ✅ 已选中** | 拦截 m3u8 playlist 请求，解密 key 后以 `data:` URI 内联到 playlist，HLS.js 原生处理 | 小 | 低 | 复用现有 WASM 代码，只改 loader 拦截点 |
| 方案二 | Cloudflare Worker 层透明解密 key，返回标准 AES 密钥 | 中 | 中 | 需在 Worker 中移植/加载 WASM，部署验证 |
| 方案三 | 手动 fetch + TS 片段解密 + MediaSource `appendBuffer`，完全绕过 HLS.js | 大 | 高 | 需自行实现 playlist 解析、segment 调度、MSE 管理 |

### 方案一：m3u8 内联 data: URL key（选中的实现方案）

#### 原理

在 HLS manifest 中，`#EXT-X-KEY` 的 URI 可以指向任意 URL，包括 `data:` URL。将解密后的 AES-128 密钥以 base64 编码嵌入 `data:` URL，HLS.js 原生加载该 URL 得到密钥，然后用其解密 TS 片段。

```
原始 m3u8:  #EXT-X-KEY:METHOD=AES-128,URI="enc_10620278.key"
修改后:     #EXT-X-KEY:METHOD=AES-128,URI="data:application/octet-stream;base64,vGEFuhkFrG46+bdxx8+apw=="
```

#### 实现步骤

| 步骤 | 说明 |
|:----:|:-----|
| 1 | 自定义 HLS loader，仅拦截 `type === 'manifest'` 或 `type === 'level'` 且 URL 包含 `.m3u8` 的请求 |
| 2 | `fetch` 获取 m3u8 文本 |
| 3 | 正则匹配 `URI="xxx.key"`，解析出 `.key` 的绝对 URL |
| 4 | `fetch` 获取 `.key` 文件（被包装的 16 字节密钥） |
| 5 | 用已有 `decryptWasmKey(wasm, wrapKey, keyBuf)` 解密 |
| 6 | 将解密后的密钥 base64 编码，替换 URI 为 `data:application/octet-stream;base64,${b64}` |
| 7 | 通过 `callbacks.onSuccess` 返回修改后的 m3u8 文本 |
| 8 | HLS.js 正常解析 playlist，加载 `data:` URI 得到密钥，解密 TS 片段 → 播放 |

#### 优势

- **无需自定义 key loader**：密钥以标准 `data:` URL 呈现，HLS.js 内部 key loading 流程完全不变
- **无需关心 loader 回调签名兼容性**：playlist loader 的 `onSuccess` 签名简单稳定
- **复用现有代码**：`getWasmInstance` / `getWrapKey` / `decryptWasmKey` 全部直接可用
- **只需修改一处**：替换 `WasmKeyLoader` 为 `WasmM3u8Loader`，拦截条件从 `.key` 改为 `.m3u8`

### 方案二：Cloudflare Worker 透明解密（备选）

#### 原理

在 Cloudflare Worker 的请求转发层，检测 m3u8 响应的 `#EXT-X-KEY`，自动完成 WASM 解密并替换为内联密钥，客户端无需任何改造。

#### 实现步骤

1. Worker 转发 m3u8 请求时，读取响应体
2. 检测 `#EXT-X-KEY:URI="xxx.key"`
3. 发起额外请求获取 `.jpg` 和 `.key` 文件
4. 在 Worker 中加载 `jquery.wasm` 并执行解密
5. 替换 m3u8 中的 URI 为 `data:` URL
6. 返回修改后的 m3u8

#### 注意事项

- Cloudflare Workers 支持 WebAssembly（`WebAssembly.instantiate()`），但 Emscripten 生成的 WASM 依赖 `env` import 对象需要模拟
- 每次 m3u8 请求都需要额外 2-3 次上游请求（`.jpg`、`.key`、`.wasm`），增加延迟
- 建议将 `jquery.wasm` 预加载到 Worker 全局变量，避免重复下载

### 方案三：MediaSource 直接播放（备选）

#### 原理

完全绕过 HLS.js，自行实现：
1. fetch m3u8 → 解析片段列表
2. fetch `.jpg` → 包装密钥
3. fetch `.key` → WASM 解密
4. fetch TS 片段 → AES-128-CBC 解密
5. MediaSource `sourceBuffer.appendBuffer(decodedTS)`

#### 复杂度

| 模块 | 实现量 | 风险 |
|:-----|:------:|:----:|
| m3u8 parser | ~100 行正则解析 | 低（格式简单） |
| AES-128-CBC 解密 | 使用 Web Crypto API | 中（IV 提取） |
| segment 调度 | ~200 行 | 中（时序控制） |
| MSE buffer 管理 | ~100 行 | 高（边界条件多） |
| 错误恢复 | 难以覆盖 | 高 |

## 1. 背景

`src/utils/image.ts` 中的 `loadImg()` 目前使用 `Promise.all` 并发发起所有图片解码请求，存在以下问题：

| # | 问题 | 影响 |
|---|------|------|
| 1 | 无并发控制，瞬间发起全部请求 | 大量图片时占满浏览器连接池，阻塞 API 请求 |
| 2 | 无缓存，同一 URL 重复解码 | 冗余网络请求 + 重复 Base64 运算 |
| 3 | 重复的 IntersectionObserver | `headicon.ts` 和 `content.ts` 各有一个，逻辑重复 |
| 4 | 无重试，错误静默吞掉 | 偶发网络失败直接显示原始 URL（无效） |
| 5 | 无优先级，可见图/不可见图同等对待 | 首屏渲染慢 |
| 6 | 无进度通知 | 调用方无法感知加载状态 |

## 2. 设计目标

- **队列化**：有限并发 + 优先级调度 + 指数退避重试
- **共享**：统一 IntersectionObserver 实例
- **缓存**：内存去重，相同 URL 只请求一次
- **可观测**：暴露加载进度和队列状态

│  └────────────────────────────────────┘   │
│  ┌────────────────────────────────────┐   │
│  │         AsyncQueue                 │   │  优先级队列 + 并发限制
│  │  ┌──────┐ ┌──────┐ ┌──────┐       │   │
│  │  │high  │ │medium│ │low   │       │   │  3 个优先级 bucket
│  │  └──────┘ └──────┘ └──────┘       │   │
│  │  maxConcurrency: 6 (可配)          │   │

## 5. 接口定义

### 5.1 导出类型

```typescript
export interface LoadOptions {
  priority?: 'high' | 'medium' | 'low'
  concurrency?: number
  signal?: AbortSignal
}

export interface LoadCallbacks {
  onProgress?: (completed: number, total: number, failed: number) => void
  onEach?: (url: string, result: string | null) => void
}

export interface QueueStatus {
  pending: number
  loading: number
  completed: number
  failed: number
  cached: number
}
```

### 5.2 导出对象

```typescript
export const imageLoader: {
  /** 批量加载图片，返回 url → decoded data URI 的 Map */
  load(urls: string[], options?: LoadOptions & LoadCallbacks): Promise<Map<string, string>>

  /** 通过 IntersectionObserver 观察元素，进入视口时自动加载 */
  observe(el: HTMLImageElement, url: string): void

  /** 停止观察 */
  unobserve(el: HTMLImageElement): void

  /** 当前队列状态 */
  getStatus(): QueueStatus

  /** 取消指定 URL 的待处理任务（不传参则取消全部 pending） */
  cancel(urls?: string[]): void

  /** 清空内存缓存 */
  clearCache(): void
}
```

### 5.3 内部类型

```typescript
interface ImageTask {
  id: string
  url: string
  priority: number          // 高=3, 中=2, 低=1
  status: 'pending' | 'loading' | 'completed' | 'failed'
  result: string | null
  retryCount: number
  maxRetries: number
  createdAt: number
  resolve?: (result: string) => void
  reject?: (error: Error) => void
}
```

## 6. 核心实现逻辑

### 6.1 AsyncQueue 并发控制

- 维护 3 个优先级数组：`high[]`、`medium[]`、`low[]`
- `push()` 按优先级入队，返回 `Promise<string>`
- `process()` 每次检查 `activeCount < maxConcurrency`，从高优先级数组中取任务执行
- 任务执行完毕递归调用 `process()`

### 6.2 内存缓存

- `Map<url, Promise<string>>`，key 为原始 URL
- 命中缓存直接返回已有 Promise，不重复请求
- 加载失败后从缓存中删除（允许下次重试）
- 不持久化到 localStorage（Base64 data URI 体积过大）

### 6.3 重试策略

```
第 1 次失败 → 等待 1000ms → 重试
第 2 次失败 → 等待 2000ms → 重试
第 3 次失败 → 等待 4000ms → 重试
第 4 次失败 → 标记 failed，返回 null
```

### 6.4 IntersectionObserver

- 统一单例 observer，`rootMargin: '300px 0px'`
- 可见时按优先级处理：
  - `isIntersecting === true` → high
  - `rootBounds` 附近 300px → medium
  - 其余 → low
- 加载完成后 `unobserve`

### 6.5 元素生命周期管理

离开页面后自动清理未完成的加载任务，避免资源浪费。

#### 6.5.1 WeakMap 元素追踪

`ImageLoader` 内部维护 `observedElements: WeakMap<HTMLImageElement, string>`：

- `observe(el, url)` 时写入映射
- `unobserve(el)` 时删除映射
- 元素从 DOM 移除时，WeakMap 自动允许 GC
- observer 回调 `handleIntersection` 通过 WeakMap 获取 URL，而非依赖 `dataset`

#### 6.5.2 自动取消逻辑

- 元素可见后开始加载 → `inflight` 标记已进入执行阶段 → 即使元素后续被销毁也不再取消（执行中无法中断）
- 元素尚未可见即被销毁 → `unobserve()` 从 observer 移除，不做入队操作，无浪费
- 已在队列中 pending 但尚未开始执行的任务 → **不会发生**，因为入队发生在 observer 触发时，而不是 observe 时

#### 6.5.3 指令 unmounted 钩子

两个 Vue 指令均添加 `unmounted` 生命周期钩子：

| 指令 | 清理操作 |
|:----|:---------|
| `v-headicon` | 查找容器内 `<img>`，调用 `imageLoader.unobserve()` |
| `v-content` | 遍历所有 `img:not([data-emoji])`，逐个调用 `imageLoader.unobserve()` |

```typescript
// v-content unmounted 示意
unmounted(el) {
  el.querySelectorAll('img:not([data-emoji])').forEach(img => {
    imageLoader.unobserve(img)
  })
}
```

## 1. 背景

通过深入分析 origin 代码（`docs/reference/origin/app.js`），发现两处关键差异：

### 问题一：`_preview.m3u8` 视频仅为 30 秒

服务端通过 `POST /api/attachment` 返回的 `remoteUrl` 可能是 `_preview.m3u8`（30秒预览文件），也可能返回完整视频。区别在于请求参数 `line`：

| `line` 值 | 说明 |
|:----------|:------|
| `""`（空字符串） | 默认线路，origin 初始加载使用 |
| `"normal1"` | 线路一（当前项目硬编码，可能返回 preview）|
| `"normal2"` | 线路二 |
| `"vip"` | VIP 线路 |

origin 通过 `GET /api/topic/att/{attachmentId}` 获取可用线路列表，用户可切换不同 `line` 值获取不同 `remoteUrl`。

### 问题二：HLS keyPath 片段路径缺失

origin 的 `customHls` 中在 `MANIFEST_PARSED` 事件里用 `key-path` 属性修正 HLS 片段的相对路径和加密密钥 URL。当前项目完全缺失此逻辑，可能导致部分视频片段加载失败。

### 问题三：`_preview.m3u8` 返回的只是 30 秒预览文件

测试发现所有 `line` 参数（`normal1`/`normal2`/`vip`）均返回 `_preview.m3u8`。参考 `docs/reference/haijiao_download/haijiao.py:168-180`，完整视频的获取方式是通过解析 preview m3u8 文件内容，从第一个 `.ts` 片段 URL 中提取 code，替换原 URL 中的 `\d+_i_preview` 得到完整视频的 m3u8 地址。

### 问题四：30 秒预览截断（已修复）

origin 中预览截断是纯客户端行为（`data-preview="30"` + `timeupdate` 事件），v1.0.0 已完成移除。

### 4.5 keyPath 值 `"undefined"` 过滤

服务端返回的 `key-path` 属性值可能是字符串 `"undefined"`，直接拼接会导致 URL 出现 `undefinedhttps://...`。读取时需过滤：

```typescript
const raw = videoDiv?.getAttribute('key-path') || ''
const keyPath = (raw === 'undefined' || !raw) ? '' : raw
```

### 4.6 resolveRealUrl：从 preview 获取完整视频 URL

参考 `docs/reference/haijiao_download/haijiao.py:168-180`：

```typescript
export async function resolveRealUrl(previewUrl: string): Promise<string> {
  const resp = await fetch(previewUrl)
  const text = await resp.text()
  const lines = text.split('\n').filter(l => !l.startsWith('#') && l.trim())
  if (!lines.length) return previewUrl
  const firstLine = lines[0]
  let code: string
  if (firstLine.startsWith('http')) {
    code = firstLine.slice(firstLine.lastIndexOf('/') + 1, firstLine.lastIndexOf('.') - 1)
  } else {
    code = firstLine.slice(0, firstLine.lastIndexOf('.') - 1)
  }
  return previewUrl.replace(/(\d+_i)_preview/, code)
}
```

### 4.7 HLS keyPath 修正

```typescript
customHls: (video: HTMLVideoElement) => {
  const hls = new Hls()
  if (keyPath) {
    hls.on(Hls.Events.MANIFEST_PARSED, (_event, data) => {
      const fragments = data.levels[0]?.details?.fragments
      if (!fragments?.length) return
      const f0 = fragments[0] as any
      const keyUri = keyPath + (f0.levelkey?.reluri || f0.levelkeys?.['identity']?.uri || '')
      for (const frag of fragments) {
        const f = frag as any
        if (f.levelkey) f.levelkey.reluri = keyUri
        else if (f.levelkeys?.['identity']) f.levelkeys['identity'].uri = keyUri
        if (f.relurl && !f.relurl.startsWith('http')) f.relurl = keyPath + f.relurl
      }
    })
  }
  hls.loadSource(video.src)
  hls.attachMedia(video)
}
```

### 4.8 播放流程

```
用户点击视频缩略图 img
  → 读取 key-path → 过滤 "undefined"
  → loadVideoSrc(id, topicId, 'normal1') → remoteUrl (带 _preview)
  → resolveRealUrl(remoteUrl) → fullUrl (完整视频)
  → DPlayer + HLS (含 keyPath 修正)
```

### 4.9 关于线路按钮和测试按钮

所有 `line` 参数（`normal1`/`normal2`/`vip`）均返回 `_preview.m3u8`，通过 `resolveRealUrl` 统一解析为完整视频。因此移除 UI 上的线路选择按钮和测试按钮，视频播放固定使用 `'normal1'` 线路。`loadVideoSrc()` 的 `line` 参数、`getVideoLines()` API 和 `api.videoLines()` 的代码保留，后续如需恢复线路切换只需还原 UI 渲染部分。

### 4.10 sell 内容视频预加载

sell-btn HTML 中可能包含 `<video data-id="xxx">` 标签，但 `attachList` 中没有对应的 `category: 'video'` 附件，导致嵌套缩略图缺失。

**处理流程：**

```
sell-btn 替换前
  → 扫描 content 中 sell-btn 块内的 <video data-id="xxx">
  → 如果 attachList 中没有该 videoId 的 video 附件
    → 调用 loadVideoSrc(videoId, topicId, '') 获取视频元数据
    → 缓存到 sellVideoFallback
  → 执行 sell-btn 替换：
    → 先从 attachList 找 videoAtta
    → 没找到则用 sellVideoFallback
    → 生成 hv-video-div 嵌套进 sell 容器
  → el.innerHTML = content + 后续 DOM 操作
```

## 5. 边界情况

- 无线路列表或仅 1 条线路 → 不渲染按钮，退化为默认行为
- API 请求失败 → 静默兜底，不影响视频播放
- 多个视频附件 → 每个独立渲染自己的线路按钮
- 出售帖（sell）内的视频 → 同样渲染线路按钮

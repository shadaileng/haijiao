## 1. 背景

### 当前代理架构

| 层级 | 代码位置 | 当前值 | 是否可配置 |
|:-----|:---------|:-------|:----------|
| 前端 API_BASE | `src/api/request.ts:4` | `'/api'`（硬编码） | ❌ |
| Vite 开发代理 | `vite.config.ts:15` | `'https://haijiao.com'`（硬编码） | ❌ |
| Worker 环境变量 | `wrangler.toml:14` | `'https://haijiao.com'`（部署时设置） | 部署时可配 |
| 前端直连数据源 | `getApiHost()` | 从 `proxyBase` 推导，回退 `'haijiao.com'` | ❌ 由 proxyBase 统一接管 |

### 问题

`API_BASE = '/api'` 硬编码在 `request.ts` 中，所有 API 请求只能通过同域 Worker（或 Vite 开发代理）转发。当用户使用自定义部署的 Worker 或第三方代理服务时，无法将请求指向其他地址。

### 触发场景

- 用户部署了自己的 Worker 到自定义域名
- 默认代理地址因网络原因不可用，需要切换到备用地址
- 开发阶段需要将 API 指向其他环境

### 关键约束：登录页也需要暴露配置入口

登录页（`LoginView`）是路由守卫中的公开页面（`publicPages`），未登录用户只能访问登录页和设置页。如果默认代理地址不可用，用户无法登录、也无法进入设置页修改，因此 **登录页必须提供代理地址配置入口**。

## 2. 设计方案

### 2.1 数据模型

在 `user` store 中新增 `proxyBase` 字段：

| 字段 | 类型 | 默认值 | 说明 |
|:-----|:----|:-------|:-----|
| `proxyBase` | `string` | `''` | 代理地址；空串表示使用 `/api` |

持久化存储在 `localStorage` 的 `haijiao_user` 对象中。

### 2.2 请求层逻辑变更

将硬编码的 `const API_BASE = '/api'` 改为动态获取函数 `getProxyBase()`：

```
getProxyBase() → store.proxyBase 有值 ? 使用该值 : 回退 '/api'
```

`buildProxiedUrl()` 和 `login()` 中的前缀统一改为调用 `getProxyBase()`。

### 2.3 UI 交互

使用 `<van-dialog>`（非 `showDialog`），因其支持自定义 slot 内嵌 `<van-field>` 输入框。

**登录页**：在「手动配置 UID / Token」按钮下方新增「配置代理地址」按钮 → 弹出对话框 → 输入保存。（保留，未登录场景必须）

**设置页**：替换原有的"数据源"字段为"代理地址"配置，复用同一个对话框组件。

**首页**：移除代理地址入口（已迁移到设置页）。

### 2.4 数据源统一

原 `apiBase`（`store` 中的 `'haijiao.com'`）用于图片/视频直链域名，现合并到 `proxyBase`：

- `proxyBase` 为完整 URL 时（如 `https://mirror.xyz`），资源直链域名取 `URL(proxyBase).hostname` → `mirror.xyz`
- `proxyBase` 为空或 `/api` 时，资源直链域名回退 `'haijiao.com'`
- 删除 `apiBase` 字段、`setApiBase()` 方法及其持久化逻辑

### 2.5 共享逻辑

创建 `src/composables/useProxyConfig.ts`，封装代理地址的读写和对话框状态，供两个页面复用。

  const proxyUrl = ref(store.proxyBase)
  const showDialog = ref(false)
  const proxyDisplay = computed(() => store.proxyBase || '/api（默认）')

  function openConfig() { ... reset proxyUrl, showDialog = true }

  return { proxyUrl, showDialog, proxyDisplay, openConfig, saveConfig }

## 4. 数据流

```
用户输入代理地址
    → store.setProxyBase(url)
        → localStorage ('haijiao_user' 对象)
    → request.ts 下次请求自动读取
        → getProxyBase()
            → store.proxyBase 有值 ? 配置值 : '/api'
    → buildProxiedUrl() / login() 使用新地址发起请求
```

## 7. 验证方法

| 验证项 | 方法 | 预期 |
|:-------|:-----|:------|
| 默认行为不变 | 不配置代理地址，请求 API | 仍走 `/api` 路径 |
| 配置后生效 | 输入 `https://custom-proxy.com`，查看网络请求 | 请求发往 `https://custom-proxy.com/api/...`（自动追加 `/api` 前缀） |
| 清空后回退 | 清空代理地址 | 请求回退到 `/api` |
| 持久化 | 刷新页面后查看 | 代理地址值从 localStorage 恢复 |
| 登录页配置 | 未登录状态配置代理地址 | 登录请求发往新地址 |
| 自定义地址 CORS | 输入 `https://custom-worker.com` 后调用 API | 请求发往 `https://custom-worker.com/api/...`（带 `/api` 前缀） |

## 8. v1.1.0 Bug 修复

### 8.1 CORS 错误：自定义代理地址缺少 `/api` 前缀

**问题**：当 `proxyBase` 设为完整 URL（如 `https://hj260601ced.top`）时，请求变成 `https://hj260601ced.top/login/signin`。但 Worker 路径规则要求 `/api/` 前缀，直接请求目标地址会因缺少 CORS 头而被浏览器拦截。

**修复**：在 `request.ts` 中新增 `getApiPrefix()` 函数，当 `proxyBase` 是完整 URL 时自动补上 `/api`：

```
proxyBase = 'https://custom.com'  →  prefix = 'https://custom.com/api'
proxyBase = '' / '/api'           →  prefix = '/api'
```

`buildProxiedUrl()` 和 `login()` 的 URL 构造改用 `getApiPrefix()`。

### 8.2 Vant 4.9 `Toast()` 迁移

**问题**：Vant 4.9+ 废弃 `Toast()` 函数式调用，改为 `showToast()` / `showSuccessToast()`。全项目 19 处 `Toast(msg)` 和 8 处 `Toast.success(msg)` 在 v4.9.7 中不可用，会抛出 `TypeError: Toast is not a function`。

**修复**：全项目替换：

| 旧 | 新 | 数量 |
|:---|:---|:----:|
| `import { Toast } from 'vant'` | `import { showToast, showSuccessToast, showFailToast } from 'vant'` | — |
| `Toast(msg)` | `showToast(msg)` | 19 |
| `Toast.success(msg)` | `showSuccessToast(msg)` | 8 |
